# Passport Seva RAG Chatbot Implementation Guide

This document explains the current implementation of the Indian Passport Service RAG chatbot, detailing the structure and logic of the system based on the provided codebase.

## 1. System Architecture

The project is built around a Retrieval-Augmented Generation (RAG) pipeline to fetch answers from passport-related documents without requiring fine-tuning of a Large Language Model (LLM). It features a modular architecture, broken down into ingestion, retrieval, generation, intent routing, and UI.

### Key Components:
*   **UI/Frontend:** Streamlit (`app.py`, `ui/`)
*   **Data Ingestion:** Document parsing and chunking (`rag/loader.py`, `rag/chunker.py`)
*   **Vector Database:** ChromaDB (`rag/vectorstore.py`)
*   **Embeddings:** Sentence Transformers (`rag/embedder.py`)
*   **Retrieval Engine:** Hybrid Search combining Dense & BM25 (`rag/retriever.py`)
*   **LLM Generator:** Groq API using Llama-3 (`rag/generator.py`)
*   **Support Agents:** Intent router and Language Translator (`agents/router.py`, `translation/translator.py`)

## 2. Detailed Module Breakdown

### 2.1. Streamlit Interface (`app.py` & `ui/`)
The entry point is `app.py`. It uses Streamlit to render a conversational chatbot interface.
*   **Session Management:** Maintains chat history, current language preference (English/Hindi), and conversation memory (`SessionMemory`).
*   **UI Elements:** Provides a sidebar with system stats, quick-topic buttons (e.g., "Tatkal", "Renewal"), and an expander to mock document uploads.
*   **Styling:** Uses custom CSS to create distinct "user bubbles" and "bot bubbles" for a modern chat experience.
*   **Initialization:** The `load_rag_pipeline` function is heavily cached (`@st.cache_resource`) to load documents, chunk them, and prepare the vectorstore only once on startup.

### 2.2. Document Processing (`rag/loader.py` & `rag/chunker.py`)
*   **`DocumentLoader`:** Accepts one or more source directories and ingests `.txt`, `.jsonl`, and `.pdf` files from each. The system currently scans two directories:
    1.  **`data/raw/`** — curated text guides and QA pairs (e.g., `fees_structure.txt`, `tatkal_process.txt`, `curated_qa.jsonl`).
    2.  **`passport_docs/`** — official government PDF documents including:
        *   `140_The-Passport-Act-1967.pdf`
        *   `AnnexureH.pdf`
        *   `ApplicationformInstructionBooklet-V3.0.pdf`
        *   `CBT_0120978_Y.pdf`
        *   `GUIDELINES FOR CAPTURING PHOTOGRAPHS FOR MINORS_v2.1.pdf`
        *   `Pasport_servic_Final.pdf`
        *   `Passport_Rules_1980.pdf`
        *   `lu1943_1.pdf`
    For PDFs, it relies on `pypdf` to extract text page-by-page. It applies basic text cleaning (removing excess whitespace and newlines).
*   **`SmartChunker`:** Splits the loaded documents into smaller, manageable chunks suitable for embeddings. Instead of hard character cuts, it intelligently splits text on sentence boundaries (`. ! ?`) ensuring the semantic meaning of chunks remains intact. It supports overlaps to maintain context between adjacent chunks.

### 2.3. Embeddings & Storage (`rag/embedder.py` & `rag/vectorstore.py`)
*   **`EmbeddingEngine`:** Utilizes the `sentence-transformers` library to convert text chunks into dense numeric vectors (384-dimensional arrays).
*   **`PassportVectorStore`:** Wraps a `chromadb.PersistentClient`. It indexes chunk embeddings alongside metadata such as `source` and `category`. During searches, it computes cosine distance to find the closest matching chunks for a user's query.

### 2.4. Hybrid Retrieval (`rag/retriever.py`)
*   **`HybridRetriever`:** Implements an advanced retrieval strategy that combines two methods:
    1.  **Dense Retrieval:** Using the vector embeddings from ChromaDB to find semantically similar text.
    2.  **Sparse Retrieval (BM25):** Using keyword-based matching via `rank_bm25` to catch exact terminology.
*   It normalizes and combines the scores from both methods using an `alpha` weighting parameter (e.g., 0.7 for Dense, 0.3 for BM25), resulting in highly accurate document retrieval.

### 2.5. Intent Routing (`agents/router.py`)
*   **`IntentRouter`:** Uses regular expressions to classify the user's query into predefined categories like `eligibility`, `documents`, `fees`, `tatkal`, etc. 
*   This intent can then be mapped to a ChromaDB metadata filter, optionally restricting the search to only chunks within that specific category to minimize hallucinations and irrelevant results.

### 2.6. Bilingual Support (`translation/translator.py`)
*   **`BilingualTranslator`:** Uses `deep_translator` to provide a seamless Hindi-English experience.
*   If the user asks a question in Hindi, the system first translates it to English before querying the vector database (since the documents are primarily in English). 
*   Once the English answer is generated by the LLM, the translator flips it back into Hindi before displaying it in the UI.

### 2.7. Response Generation (`rag/generator.py`)
*   **`ResponseGenerator`:** Integrates with the **Groq API**, utilizing a lightweight and incredibly fast model (like `llama-3.3-70b-versatile`).
*   It constructs a final prompt encompassing:
    1.  The recent conversation history.
    2.  The retrieved document chunks (Context).
    3.  The user's specific question.
*   The system prompts the LLM to strictly answer *only* based on the context and to append the source file names for transparency.

## 3. The Execution Flow

1.  **User Input:** The user types a query (e.g., "How much is the fee for a Tatkal passport?") in the Streamlit UI.
2.  **Translation (Optional):** If the query is in Hindi, it translates it to English.
3.  **Intent Classification:** The system determines the query's intent to apply optional metadata filters.
4.  **Retrieval:** The query is embedded, and the `HybridRetriever` fetches the top `N` relevant chunks using both Dense and BM25 techniques.
5.  **Generation:** The LLM receives the prompt with the chunks and history and generates a grounded answer.
6.  **Translation & Output:** The answer is translated back to Hindi (if needed) and displayed in the chat interface alongside the referenced PDF source.
